## 0. 设计原则（开发前必读）

### 0.1 这个工具管什么

**管进度线，不管任务清单。**

这是与飞书、Notion、滴答清单的根本区别。

- ✅ 工作线是一个"持续推进的事"，比如"张三（博士）"、"国自科面上"、"新房装修"
- ✅ 每次更新只记录"这条线现在到哪了"，一句话即可
- ❌ 不是把这件事拆成20个子任务，然后逐条打勾
- ❌ 不是项目管理软件，不需要甘特图、依赖关系、负责人分配

**判断标准**：如果你在想"要不要把这个再拆成几个子任务"，那你已经走偏了。一条工作线 = 一个你需要持续关注进度的事。

### 0.2 复杂度控制

- 全部工作线不超过 30 条（超出要合并或归档）
- 每次更新的状态描述控制在 1-2 句话
- 界面不超过 3 层深度（首页 → 角色视图 → 历史展开）
- 功能不做任务分解、不做子项目、不做评论系统

## 1. 产品定位

**一句话定义**：一个按角色组织的个人工作线看板，帮你每天用 1 分钟确认哪些线在动、哪些线停了、今天主动推哪几件。

**核心价值**：从"被事情推着走"变成"每天有意识地推动每条线"，防止任何重要的事情在不知不觉中停滞一个月。

**使用频率**：每天上班打开，花 1-2 分钟看一眼；有进展时随时更新（包括出差途中用手机）。

## 4. 数据模型（每条工作线的字段）

```json
{
  "id": "uuid",
  "role": "R1",
  "name": "218A 项目",
  "priority": "高",
  "currentStatus": "经费已报，等批复",
  "lastUpdateTime": "2024-01-15",
  "statusLight": "green",
  "archived": false,
  "history": [
    {
      "time": "2024-01-15",
      "type": "manual",
      "content": "经费已报，等批复"
    },
    {
      "time": "2024-01-08",
      "type": "manual",
      "content": "催了一遍财务，本周处理"
    },
    {
      "time": "2024-01-01",
      "type": "system",
      "content": "停滞预警（8天未更新）"
    }
  ]
}
```

**字段说明：**

|字段|类型|说明|来源|
|---|---|---|---|
|id|string|UUID，唯一标识|系统生成|
|role|enum|R1/R2/R3/R4|新建时选择|
|name|string|工作线名称，如"李兴明（博士）"|用户填写|
|priority|enum|高/中/低|用户手动设置，默认"中"|
|currentStatus|string|最新一次更新的内容，1-2句话|每次更新写入|
|lastUpdateTime|date|最后一次手动更新的日期|自动记录|
|statusLight|enum|red/yellow/green|系统按规则计算|
|archived|bool|是否归档|用户操作|
|history|array|所有更新记录（含系统预警）|累积|

**状态灯计算规则：**

|颜色|条件|
|---|---|
|🟢 绿|最近 5 天内有手动更新|
|🟡 黄|6-10 天未手动更新|
|🔴 红|11 天以上未更新，或新建且未填初始状态|

> 注：系统自动写入的"停滞预警"不刷新状态灯，只有手动更新才算。

### P0：核心功能（必须有）

#### 5.1 首页概览

**今日必推（首屏核心）**

#### 5.3 批量更新（语音/文本解析）

**入口**：首页底部和每个角色视图底部，固定悬浮按钮"📝 批量更新"。

**流程：**

1. 点击 → 弹出全屏输入框（手机上占满屏幕）
2. 输入框顶部有示例提示文字（灰色占位）：

    > "说说今天的进展，比如：孙启尧今天发了初稿给我。218A项目的经费审批下来了。装修这周没动。"

3. 语音输入（手机输入法）或键盘输入
4. 点击"解析" → 调用 AI API → 进入预览确认页

**预览确认页：**

|匹配|工作线|当前状态|AI解析内容|置信|操作|
|---|---|---|---|---|---|
|✅|⭐ [R4] 孙启尧|停滞中|发了初稿给我|高|[✓] [编辑]|
|✅|⭐ [R1] 218A项目|审批中|经费审批下来了|高|[✓] [编辑]|
|⚠️|[R2] 装修|设计阶段|这周没动|中|[✓] [编辑] [忽略]|

未提及的工作线集中在底部，停滞的红色线单独高亮，提供批量操作： [全部保持现状] [全部标记今日无进展]

底部：[全部采纳] [确认更新] [取消]

#### 5.4 工作线管理

- **新增**：[+新增] 弹窗，填名称、选角色、选优先级、填初始状态（选填）
- **编辑**：点击名称旁的编辑图标，可修改名称、角色、优先级
- **归档**：右侧 [归档] 按钮，移入归档区（不删除）
- **恢复**：归档视图中可恢复，历史记录完整保留
- **删除**：永久删除需二次确认，一般只用归档

### 6.3 手机端布局（必须适配）

**导航**：左侧导航在手机上折叠为底部 Tab 栏（5个 Tab：首页、R1、R2、R3、R4；归档/年报放入"更多"）

**列表**：工作线卡片占满屏幕宽度，信息分两行显示：

```
行1：⭐ 孙启尧（博士）             🔴 12天前
行2：停滞中，上次说在修数据         [更新]
```

**批量更新按钮**：固定在底部 Tab 栏中央，圆形大按钮（≥ 56px），视觉突出。

**输入框**：点击后全屏展开，适配手机键盘弹起，字体不小于 16px（防止 iOS 自动缩放）。

**单条更新**：点击 [更新] 后弹出底部抽屉（bottom sheet），不是弹窗（手机上弹窗体验差）。

**时间轴展开**：点击后在当前页面内联展开，不跳转新页面。

## 7. 非功能需求

### 7.1 技术栈

|组件|方案|说明|
|---|---|---|
|前端|原生 HTML/CSS/JS（或 Vue 3 CDN）|单文件可维护，无构建流程|
|数据存储|GitHub API + 私有仓库 data.json|跨设备同步，数据在自己手中|
|本地缓存|localStorage|离线草稿，联网后同步|
|AI 解析|智谱 GLM-4 或 GPT-4o-mini|函数调用模式，结构化输出|
|部署|GitHub Pages|免费，手机电脑同一 URL|

### 7.2 GitHub API 数据同步

**原理**：`data.json` 存在私有仓库，读写通过 GitHub REST API 操作。

- 读取：`GET /repos/{owner}/{repo}/contents/data.json` → Base64 解码
- 写入：先获取 `sha`，再 PUT 写入（原子替换）
- 冲突：`sha` 不匹配时提示"数据已在其他设备更新，请刷新"
- 离线：操作写入 localStorage 草稿，联网后提示同步
- 延迟：读 < 500ms，写 < 1s

### 7.3 性能

- 首次加载（含 GitHub API）：< 2s
- AI 解析响应：< 5s
- 本地渲染（30条）：< 100ms

### 7.4 安全性

- GitHub Token 存 localStorage，不写入代码
- 私有数据仓库不对外公开
- AI 调用不存储业务数据

## 8. 用户流程示例

### 早上上班（电脑端，2分钟）

1. 打开网页，直接看首页"今日必推"
2. 看到：孙启尧🔴停滞12天、218A项目🔴停滞8天、备孕🟡6天未记录
3. 决定今天：① 给孙启尧发消息催进度 ② 问一下218A经费 ③ 等晚上回家跟老婆聊备孕
4. 花30秒在看板上记录这三个待推动的想法

### 下午出差途中（手机端，1分钟）

1. 打开网页（和电脑端同一URL），数据已同步
2. 点底部中央"📝 批量更新"大按钮
3. 用手机输入法语音说：「孙启尧下午回消息了说数据还有问题，本周五交。218A的经费财务说下周审。备孕的事晚上跟老婆定了去做检查。」
4. 点解析，预览确认，点确认
5. 数据同步到 GitHub，回到办公室打开电脑状态已更新

### 一周后发现某线停滞

- 首页"今日必推"中出现：⭐ [R3] 成果商业化 🔴 停滞14天
- 你意识到创业这块已经两周没推了，马上给爱人发消息约一个对话时间
- 更新状态："约了本周六下午讨论商业化进展"

## 9. 风险与应对

|风险|应对|
|---|---|
|GitHub API 失败|写入 localStorage 草稿，显示"离线模式"标识，联网后一键同步|
|sha 冲突|提示刷新，用最新版本，极低概率发生（你是唯一用户）|
|AI 解析偏差|预览页置信度 + 人工确认，错误不直接写入|
|工作线越来越多|定期归档已完成/暂停的线，保持总量 ≤ 30 条|
|工具本身变成负担|更新控制在"一句话"，批量更新支持说一段话更新多条，降低使用摩擦|
